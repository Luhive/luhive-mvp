## Google Maps location — Stage 1 (foundation + form)

Integrated Google Maps into event creation: Places autocomplete, structured storage, and in-form map preview (Luma-style). Provider: Google Maps. Store strategy: structured fields + derive links at read time (no stored Maps URLs).

### Schema & types

- **Migration** `20260615_events_location_structured.sql` — nullable `location_name`, `location_lat`, `location_lng`, `location_place_id` on `events` (legacy `location_address` kept)
- **`database.types.ts`** — updated `events` Row/Insert/Update
- **`app/modules/events/model/event-location.types.ts`** — `LocationValue` (`name`, `address`, `lat`, `lng`, `placeId`)

### `GoogleMaps` util (`app/modules/events/utils/google-maps.ts`)

Static class, single entry point for all Maps/Places usage:

- **Loader** — dynamic `import('@googlemaps/js-api-loader')` inside `loadPlaces()` (SSR-safe; avoids `window is not defined`)
- **Places API (New)** — `AutocompleteSuggestion.fetchAutocompleteSuggestions` + `place.fetchFields()` via session tokens
- **Location bias** — default 50 km around Baku when no bias passed (fixes irrelevant results e.g. Thailand from AZ)
- **URL builders** — `embedUrl` (free Maps Embed API iframe), `mapsLink`, `directionsLink`

### Form UX

- **`use-place-autocomplete.ts`** — debounced suggestions, session token lifecycle, optional `locationBias`
- **`event-location.tsx`** — custom dropdown on Places Data API; replaces plain address input; shows `LocationMapPreview` after selection
- **`location-map-preview.tsx`** — venue header + free Embed API iframe (initially under `event-form/fields/`)
- **`event-form.tsx`** — `location: LocationValue | null` state; validation/change detection; create + edit payloads include structured fields
- **`event-create-action.server.ts`** — persists `location_name`, `location_lat`, `location_lng`, `location_place_id`
- **`event-edit.tsx`** — hydrates `initialData.location` from DB columns

### Tooling & fixes

- **`@types/google.maps`** + `"google.maps"` in `tsconfig.json` `types`
- **`VITE_GOOGLE_MAPS_API_KEY`** — Maps JavaScript API, Places API (New), Maps Embed API must be enabled in Cloud Console (Embed 403 until enabled)
- Loader v2 API: `setOptions({ key, v })` not `apiKey`; no `sessionToken` on `fetchFields`

## Custom registration questions — dropdown (Options) type

Added Luma-style **Options** questions alongside existing text questions on the event registration form. No DB migration — `custom_questions` / `custom_answers` are JSON columns; new fields are additive.

### Data model

- **`event.types.ts`** — `CustomQuestionType` (`text` | `dropdown`); `DropdownOption` (`id`, `label`); `CustomQuestion` extended with `type?`, `options?`, `allowMultiple?`; `CustomAnswerJson` values widened to `string | DropdownOption | DropdownOption[]`
- **`custom-questions-constants.ts`** — `MIN_DROPDOWN_OPTIONS`, `MAX_DROPDOWN_OPTIONS`, `MAX_OPTION_LENGTH`
- Backward compat: questions without `type` treated as `text` via `question.type ?? 'text'`

### Utils (`app/modules/events/utils/custom-questions.ts`)

- `createNewCustomQuestion`, `createNewDropdownQuestion`, `createNewDropdownOption`
- `validateCustomAnswers` — branches on question type; dropdown validates selected option IDs against configured options
- `flattenCustomAnswers` / `formatAnswerForDisplay` — CSV and display join option labels (comma-separated for multi-select)

### Stored answer shape

```json
{
  "text-question-id": "free text answer",
  "dropdown-single-id": { "id": "opt-uuid", "label": "Student" },
  "dropdown-multi-id": [
    { "id": "opt-uuid-1", "label": "Student" },
    { "id": "opt-uuid-2", "label": "Engineer" }
  ]
}
```

Option IDs are generated when the admin adds options in the builder (`crypto.randomUUID()`), so answers stay stable for future grouping even if labels change.

### Admin builder (`custom-questions-builder.tsx`)

- **Text** / **Options** add buttons (Options uses `CircleDot` icon)
- Per-question card: label, required toggle; dropdown adds options editor (add/remove rows), Single/Multiple segmented control
- Validation: dropdowns need at least one non-empty option label

### Registrant form (`custom-questions-form.tsx`)

- Text → `Textarea`
- Dropdown single → `Select` (“Select an option”); stores full `DropdownOption`
- Dropdown multiple → checkbox list; stores `DropdownOption[]`

### Display & export

- **`registration-answers-display.tsx`** — shows option labels via `formatAnswerForDisplay`
- Server action already JSON-parses `custom_answers` and calls `validateCustomAnswers` — no server changes needed

### Cleanup

- **Deleted** unused duplicates `app/shared/lib/utils/custom-questions.ts` and `app/shared/lib/constants/custom-questions.ts` (event-specific code; not imported anywhere)

### Builder UX (add question feedback)

Clicking **Text** or **Options** now: smooth-scrolls to the new card, focuses the question input, and applies a 2s primary ring highlight so it’s obvious the action succeeded without manual scrolling.
