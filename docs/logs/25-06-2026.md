## Community join host email ‚Äî Google OAuth gap

When a guest joined a community via **Continue with Google**, community owners/admins did not receive the "New member joined" email. Email signup + OTP and logged-in join already sent it.

### Root cause

Host notifications go through `sendCommunityJoinNotification` in `app/shared/lib/email.server.ts`. That was only called from:

- `app/modules/community/server/join-action.server.ts` ‚Äî logged-in join via `/api/join-community`
- `app/modules/auth/server/verify-otp-action.server.ts` ‚Äî email + OTP modal flow

Google OAuth uses a different path: `JoinCommunityForm` ‚Üí `/signup` ‚Üí Google ‚Üí `/auth/verify?code=...` handled by `app/modules/auth/server/verify-loader.server.ts`. That loader inserted into `community_members` but never triggered host emails (same gap for email-link `verifyOtp` with `token_hash` + `type=signup`).

### Fix

- **Added** `app/modules/community/server/notify-community-join.server.ts` ‚Äî shared `notifyCommunityJoin()` helper: loads community name/slug, finds owner/admin user IDs via service role, sends `sendCommunityJoinNotification` to each (best-effort, non-blocking).
- **Updated** `app/modules/auth/server/verify-loader.server.ts` ‚Äî call `notifyCommunityJoin` after successful `community_members` insert in all three join branches (new OAuth user, existing OAuth user, email-link verifyOtp).
- **Refactored** `app/modules/auth/server/verify-otp-action.server.ts` and `app/modules/community/server/join-action.server.ts` to use the same helper (removes duplicated admin lookup + email loops).

### Verify

Join a community as a new user via **Continue with Google** on a community page ‚Äî host inbox should receive the same "New member joined" email as the normal email registration flow.

## Event Statistics ‚Äî service-role API + full redesign

Moved event statistics data fetching from the browser Supabase client (RLS-filtered) to a secured API route using the service role, and redesigned the entire dashboard page into a compact, Luma-style analytics view.

### Service-role API (bypasses RLS)

Previously the statistics page used a `clientLoader` that queried `event_visits` and `event_registrations` via the browser anon client. RLS on `event_visits` only returned rows where the user is admin of the visit's `community_id` ‚Äî so co-hosted events showed split/incomplete stats.

- **Added** `app/modules/events/model/event-statistics.types.ts` ‚Äî shared types (`EventVisitStatRow`, `EventRegistrationStatRow`, `EventStatisticsPayload`, plus UI types: `TimeRange`, `ChartPoint`, `BreakdownRow`, `Summary`, `ViewModel`)
- **Added** `app/modules/events/data/event-statistics-repo.server.ts` ‚Äî service-role queries for last 30 days of visits + registrations (full `event_id` scope, no RLS filtering)
- **Added** `app/modules/events/server/assert-event-statistics-access.server.ts` ‚Äî verifies community owner/admin status + host/co-host event access
- **Added** `app/modules/events/server/api-event-statistics.server.tsx` ‚Äî secured API loader (session auth ‚Üí authz ‚Üí service-role fetch ‚Üí JSON response)
- **Added** `app/routes/api/events/event-statistics.tsx` ‚Äî thin route re-export
- **Updated** `app/routes.ts` ‚Äî registered `api/events/event-statistics`
- **Updated** `app/routes/dashboard/event-statistics.tsx` ‚Äî `clientLoader` now fetches the API instead of using the browser Supabase client; handles 401 with login redirect
- **Deleted** `app/modules/events/data/event-statistics-repo.client.ts`

### UI redesign ‚Äî compact Luma-style layout

Replaced the bloated 461-line route (inline helpers, 4 large stat cards, area chart, 3 breakdown cards) with a clean component-based architecture.

#### Architecture

- `app/modules/events/utils/event-statistics.ts` ‚Äî pure helpers (`startOfDay`, `toDayKey`, `getDays`, `inRange`, `buildChartData`, `buildBreakdown`, `countVisitsSince`)
- `app/modules/events/hooks/use-event-statistics.ts` ‚Äî owns `range` state and all derivations (fixed-window page views 24h/7d/30d + range-filtered stats)
- `app/modules/events/components/event-statistics/`:
  - `statistics-header.tsx` ‚Äî "‚Üê Back to events" link + event cover thumbnail + event name
  - `statistics-views-chart.tsx` ‚Äî bar chart (Page Views + Registrations) with period `Select` dropdown, inline legend, and footer stat strip (30d/7d/24h | Unique/Registered/Conversion with vertical divider)
  - `statistics-breakdowns.tsx` ‚Äî Sources/Countries/Cities in 3-column grid with percentage bars and info tooltip on Top Sources

#### Key UI decisions

- **Bar chart** replaces area chart; brand primary color (`--primary`) for views, lighter tint for registrations
- **Period dropdown** (`Select`) replaces `ToggleGroup` toggle buttons
- **Footer stat strip** is left-aligned flex with a vertical divider separating page-view windows from core stats; "Registered" is green, "Conversion" uses brand primary
- **Breakdowns** show percentage bars (brand-colored) under each row for visual density
- **Info tooltip** on "Top Sources" heading: "Create a custom tracking link by adding ?utm_source=your-link-name to your URL."
- **Removed** "Avg Time To Register" card entirely
- **Event cover** plumbed through the API (`cover_url` added to event select in repo + auth helper)

### Hourly bucketing for "Last 24 hours"

The chart previously only had day-level granularity ‚Äî selecting "Last 24 hours" showed a single bar. Now `buildChartData` detects `range === "1d"` and creates 24 hourly buckets (labeled "12 AM", "1 AM", etc.) using the viewer's local timezone, giving a proper timeline view.

### Verify

- Host/co-host admins see full combined event stats (not split by community_id)
- Unauthenticated API calls ‚Üí 401; wrong community ‚Üí 404
- Chart shows hourly bars for 24h, daily bars for 3d/7d/30d
- Period dropdown updates chart + inline core stats + breakdowns
- Page Views 30d/7d/24h in footer strip stays fixed regardless of dropdown
- Breakdowns show percentage bars + tooltip on Sources
- Route file is ~109 lines (down from 461)
