## Event reminders — 5-day and 3-day offsets

Extended the event reminder system with two new email offsets: **5 days before** and **3 days before** event start. Follows the same pattern as the existing `5-hours` addition — enum migration, form UI, send-window logic, and email copy. No new tables.

### Reminder options (full list)

| Value | Label | Send window (UTC, relative to `start_time`) |
|-------|-------|-----------------------------------------------|
| `5-days` | 5 Days Before | now + 5 days ± 30 min |
| `3-days` | 3 Days Before | now + 3 days ± 30 min |
| `1-day` | 1 Day Before | now + 1 day ± 30 min |
| `5-hours` | 5 Hours Before | now + 5 h ± 15 min |
| `3-hours` | 3 Hours Before | now + 3 h ± 15 min |
| `1-hour` | 1 Hour Before | now + 1 h ± 10 min |

### Schema

- **`database.types.ts`** — enum + `Constants.public.Enums.reminder_time` updated

### UI

- **`event-reminders.tsx`** — two new checkbox options; list reordered chronologically (furthest first): 5 days → 3 days → 1 day → 5 h → 3 h → 1 h
- Create/edit save paths unchanged — `saveEventReminders()` already persists `reminder_times[]`

### Send handler

- **`api-send-reminders.server.tsx`**
  - `REMINDER_TIMES` constant includes all 6 offsets
  - `getRemindersToSend()` — windows for `3-days` and `5-days` (±30 min, same as `1-day`)
  - `getReminderSubjectSuffix()` — subject copy: `"in 5 days"`, `"in 3 days"`
  - JSDoc on external cron contract (POST once per offset with `CRON_SECRET`)
- Dedup unchanged — `sent_reminders` unique on `(registration_id, reminder_time)`

### Email template

- **`event-reminder-email.tsx`** — `getReminderText()` cases for `"in 3 days"` / `"in 5 days"`
- **`email.server.ts`** — `reminderTime` type aligned (helper unused by cron path today)

### How sending works (unchanged architecture)

External cron POSTs to `POST /api/events/send-reminders`:

```json
{ "reminderTime": "5-days" | "3-days" | "1-day" | "5-hours" | "3-hours" | "1-hour", "secret": "<CRON_SECRET>" }
```

Handler finds **published** events in the time window, filters by configured `event_reminders.reminder_times`, sends to **approved + going** registrations via Resend, records `sent_reminders`.

Recipients: auth users (email from Supabase) or anonymous (`anonymous_email`).

### Local testing (Postman)

Verified locally via:

- **URL:** `POST http://localhost:5173/api/events/send-reminders`
- **Body:** `{ "reminderTime": "<offset>", "secret": "<CRON_SECRET>" }`
- Requires: published event with `start_time` in the matching window, reminder offset enabled, approved + going registration, `RESEND_API_KEY` set

### Production cron — cron-job.org (Option A)

Six separate cron jobs on [cron-job.org](https://cron-job.org), same schedule, different body:

| Job title | `reminderTime` |
|-----------|----------------|
| Luhive Reminders - 5-days | `"5-days"` |
| Luhive Reminders - 3-days | `"3-days"` |
| Luhive Reminders - 1-day | `"1-day"` |
| Luhive Reminders - 5-hours | `"5-hours"` |
| Luhive Reminders - 3-hours | `"3-hours"` |
| Luhive Reminders - 1-hour | `"1-hour"` |

- **URL:** `https://luhive.com/api/events/send-reminders`
- **Method:** POST, `Content-Type: application/json`
- **Schedule:** every **15 minutes** (not hourly — `1-hour` window is only ±10 min)
- **`secret`** must match Netlify `CRON_SECRET`
- Optional: stagger job start by 2 min each to avoid 6 simultaneous hits

Most runs return `reminders_sent: 0` — expected when no event falls in the window.

## Event registration — full-page overlay route

Moved native event registration off the modal/sheet and onto a dedicated, URL-addressable overlay at `/c/:communitySlug/:eventSlug/register`. Goal: Luma-style focused registration (instant open/close, shareable URL) while keeping the event page mounted underneath.

### Route architecture

- **Nested child route** under event detail in `app/routes.ts`:
  - `:eventSlug` → `event-detail.tsx`
  - `:eventSlug/register` → `event-register.tsx`
- **`event-detail.tsx`** — renders `<EventDetail />` + `<Outlet />`; hides event page with `invisible` + `aria-hidden` when register child is active
- **`event-register.tsx`** — no server loader; reads parent via `useRouteLoaderData("routes/web/event-detail")`; client-side eligibility guard with `<Navigate replace>`
- Register open/close is instant (no server round-trip on navigation); parent loader is the single source of registration state

### New / updated module files

| File | Role |
|------|------|
| `event-register-view.tsx` | Orchestrates overlay shell + `RegistrationFlow` / `CustomQuestionsForm` |
| `registration-overlay-shell.tsx` | Full-screen overlay shell (X close, centered column, title) |
| `registration-flow.tsx` | Extracted guest-step logic (email → details → questions → OTP) |
| `event-register-meta.ts` | Meta derived from parent match |
| `native-can-register-view.tsx` | Register buttons navigate via `Link` to register route |
| `custom-questions-form.tsx` | `variant="overlay"` — flat profile row, muted inputs, single-line fields |
| `layout.tsx` | `isFocusedRoute` hides nav/footer on register route |

Removed modal registration state from `event-detail.tsx` component; `EventRsvpModal` kept only for event preview sidebar.

### UI polish

- Overlay: fixed full-screen `bg-background`, `max-w-[420px]` column, "Your Info" / "Tell us more" titles
- Logged-in user with custom questions: read-only profile row on overlay
- **Focus ring clipping fix** — inner wrapper had `overflow-y-auto`, which forced horizontal clip and cut off input focus rings; removed nested scroll (outer shell scrolls) and added `px-1` on content wrapper

### Stale state after registration

**Problem:** `shouldRevalidate` blocked all GET revalidations to keep overlay instant. After registering, success paths `navigate` back to the event page (GET) → parent loader never re-ran → UI still showed "not registered".

**Fix (two parts):**

1. **Narrow `shouldRevalidate`** in `event-detail.tsx` — skip GET revalidation only when opening or closing `/register` (compare `currentUrl` / `nextUrl` pathnames); POST mutations still revalidate normally
2. **One-time revalidation on success** — success navigates pass `state: { justRegistered: true }`; event detail page calls `useRevalidator().revalidate()` once on land, then clears marker via `history.replaceState`

Updated in `event-register-view.tsx`: shared `navigateAfterRegistration()` for both `onRegisterSuccess` (logged-in POST) and `handleRegistrationSuccess` (guest/OTP flows).

**Considered and rejected:** `remix-client-cache` (SWR client loader cache) — would still require explicit invalidation after mutation and adds a dependency; native `shouldRevalidate` + `revalidate()` is sufficient.

### Overlay close performance

**Problem:** Close felt laggy — `registration-overlay-shell.tsx` waited 220ms (`setTimeout`) before navigating so an exit animation could play; opaque backdrop stayed visible; invalid Tailwind classes (`duration-220`, `duration-320`) meant transition didn't match timeout anyway.

**Fix:** Navigate immediately on close (event page already mounted behind overlay). Removed phase state machine, double `requestAnimationFrame`, and exit animation. Enter animation only — CSS `animate-in fade-in slide-in-from-bottom-1.5` (same utilities as dialog/sheet), `motion-reduce:animate-none`. Shell went from ~113 lines to ~72, no state/effects.

### Expected behavior (after fixes)

| Action | Behavior |
|--------|----------|
| Open register overlay | Instant, no loader fetch |
| Cancel / X close | Instant navigate, no fetch |
| Finish registration | Overlay closes, then one revalidation → registered state + counts update

## Event invites — end-to-end flow

Added the ability for organizers and registered attenders to invite people to an event by email. Invitee gets a link, can accept without an OTP code, and shows up in the dashboard with clear invite status.

### What users can do

- **Send invites** from the attenders dashboard, event admin view, or as a registered attender ("Invite someone" button + modal)
- **See pending invites** in the attenders table — name, "Invited by …", RSVP shows **Invite Pending**, approval shows **Not accepted**
- **Accept invite** via email link — account is created/signed in automatically; if the event has custom questions they land on the register page to complete them
- **Resend** — inviting the same email again refreshes the pending invite and sends a new email

### Attenders dashboard & table

- Table includes pending (unaccepted) invites, not just confirmed attenders
- Public attendee lists still only show verified, approved people
- After sending an invite from the dashboard, the new row appears **immediately** without refreshing the page
- Clicking a row opens detail view aligned with the table: RSVP **Invite Pending** for pending invites, **Invited By** shown, removed redundant "Registration Source" / "Invite Status" rows from the drawer

### Invite modal UX fix

- Send button was briefly re-enabling while the email was still sending (~3–4 seconds) — fixed so the whole form stays disabled with "Sending…" until the request finishes

### Community membership on accept / register

- **No custom questions + invite link:** invitee is joined to the host community automatically
- **Custom questions (invite or normal register):** "Become a member of {community}?" checkbox on the **last step** (questions page, or name/surname if no questions)
- Checkbox is **hidden** if the user is already a community member
- Normal self-registration flow uses the same last-step placement (removed duplicate checkbox on OTP step)

### Notes

- Organizer invites auto-approve (when event allows); attender invites follow the event's approval setting
- Pending invites do not count toward event capacity
- Excel export still includes invite/source columns for admins even though the UI table simplified those columns

## Event page — instant registration state (React Query + SSR hydration)

Guest registration, login/logout, and register/unregister on the event page now update the UI immediately without a hard refresh or a 4–5 second stale state. SSR and SEO are unchanged: loaders still fetch everything server-side; client interactivity is layered on top via TanStack Query with proper server hydration.

### User-visible outcomes

| Scenario | Before | After |
|----------|--------|-------|
| Guest registers via OTP overlay | Registration card updated; navbar + profile stayed logged-out until refresh | Navbar, profile, and registration card all update instantly |
| Login from event page → redirect back | Navbar still showed Sign In until refresh | Avatar/menu appears immediately |
| Logout on event page | Logout button often did nothing; event still showed "registered" | One-click logout; navbar + registration card show logged-out state |
| Cancel registration | Brief flash of "Registration deadline passed" | Register button appears immediately |
| Login as already-registered user after logout | Showed "Register" until refresh | Shows "Registered" immediately |
| Logout click | UI flickered logged-out → logged-in → logged-out | Single clean transition to logged-out |

### Architecture — two React Query caches

- **`['current-user']`** — shared navbar session (profile avatar, Sign In vs logged-in menu)
- **`['event-registration', eventId]`** — event-page slice: registration status, user context, `canRegister`, community membership

Both are seeded from server loaders via **dehydrate + `HydrationBoundary`** (TanStack's recommended SSR bridge). Components read only via `useQuery`; loaders own SSR truth.

```
Server loader → dehydrateSeed() → dehydratedState in loaderData
  → <HydrationBoundary> → browser RQ cache → useQuery() on first paint
```

Optimistic updates (register, unregister, OTP, logout) still use `setQueryData`; mutations invalidate for background reconcile via `/api/current-user` and `/api/events/registration-state`.

### Phase 1 — React Query foundation

- Installed `@tanstack/react-query`; `QueryProvider` in `app/root.tsx`
- `GET /api/current-user` — session profile from cookies (navbar)
- `GET /api/events/registration-state?eventId&communityId` — full `EventPageUserState` slice
- `useCurrentUserQuery`, `useEventRegistrationQuery`, `useRegistrationCacheUpdate`
- `event-detail.tsx` merges query data into `userData` for `EventDetail`
- `shouldRevalidate` on event detail blocks GET revalidation only when toggling `/register` overlay (instant open/close preserved)

### Phase 2 — Guest OTP session + page context

- **`verify-otp-action.server.ts`** — modal success returns `userId`, `email`, `userProfile`, `registrationState`, `isCommunityMember` when event registration completes
- **`otp-input-inline.tsx`**, **`registration-flow.tsx`**, **`event-register-view.tsx`** — full OTP result passed through; both caches updated on success
- **`fetch-event-page-user-state.server.ts`** — shared server builder for registration API and OTP response
- **`EventPageUserState`** type — registration fields + `user`, `userProfile`, `isCommunityMember`, `canRegister`

### Phase 3 — Login / logout navbar sync

- **Logout button fix** — removed broken `asChild` on `<LogOut />` icon (clicks were unreliable)
- **`layout.tsx`** — navbar reads from `useCurrentUserQuery` instead of frozen loader snapshot
- **`handleLogout`** — optimistic cache clear before POST to `/logout`
- **`clearEventRegistrationSessionState`** — logout also clears event-registration cache (navbar alone was not enough)

### Phase 4 — Cache correctness fixes

- **Unregister `canRegister`** — action returns `canRegister: true` after cancel so UI doesn't flash closed/deadline view while refetching
- **Loader → cache sync** (interim) — `useSyncEventRegistrationCache` + `useEffect` on `loaderData.userData` so login-as-registered-user after logout reflected server truth
- **Logout flicker fix** — optimistic logout uses `setQueryData` only (`revalidate: false`); no refetch while session cookie still valid mid-POST

### Phase 5 — Dehydrate + HydrationBoundary (current pattern)

Replaced manual `initialData` + sync effects with official TanStack SSR hydration.

**New shared infrastructure:**

| File | Role |
|------|------|
| `app/shared/lib/query/query-keys.ts` | Server-safe query keys (`current-user`, `event-registration`) |
| `app/shared/lib/query/dehydrate-loader.server.ts` | `dehydrateSeed()` — per-request QueryClient → dehydrated JSON |
| `currentUserQueryOptions()` / `eventRegistrationQueryOptions()` | Shared query config for hooks + loaders |

**Route changes:**

- **`layout.tsx`** — loader dehydrates profile; `<HydrationBoundary>` wraps `LayoutInner`; `useCurrentUserQuery()` inside boundary
- **`event-detail-loader.server.ts`** — builds `EventPageUserState` slice, dehydrates into `dehydratedState`
- **`event-detail.tsx`** — `<HydrationBoundary>` wraps `EventInner`; query hook with no `initialState`

**Removed (replaced by hydration):**

- `initialData` / `initialState` on query hooks
- `useSyncEventRegistrationCache` and its `useEffect`
- Layout `setQueryData` sync effect on `loaderUser`
- `refetchOnMount: "always"`

**Kept (auth/mutation complexity — inherent, not hydration):**

- `useSetCurrentUserCache(profile, revalidate?)` — OTP login, logout
- `useRegistrationCacheUpdate` — register/unregister POST, action toasts
- `clearEventRegistrationSessionState(queryClient, { revalidate? })` — logout + post-logout loader sync
- API routes for background refetch after invalidation

### Overlay / optimistic behavior preserved

`shouldRevalidate` still blocks the event-detail loader when opening/closing `/register`. Dehydrated state doesn't change on overlay toggle, so hydration does not clobber guest OTP or register optimistic cache updates.
